# 0.1.3 — 2026-06-04

**BREAKING:** `met_get_object` renames the `hasImages` output field to `hasCC0Image`. Consumers reading `result.objects[].hasImages` must update to `result.objects[].hasCC0Image`. The value is unchanged — `true` when `primaryImage` is non-empty, indicating a CC0 open-access image is available.

## Changed

- **`hasImages` → `hasCC0Image`** on `met_get_object` output ([#1](https://github.com/cyanheads/met-museum-mcp-server/issues/1)) — rename clarifies that the flag signals a CC0 open-access image, distinct from `met_search`'s `hasImages` filter which matches objects with any image including copyrighted works. Consumers reading `hasImages` must update to `hasCC0Image`.

## Added

- **`all_not_found` error contract** on `met_get_object` ([#2](https://github.com/cyanheads/met-museum-mcp-server/issues/2)) — distinguishes the case where every requested ID returns HTTP 404 (stale/invalid IDs) from network-level failures. Previously all-failure cases threw `all_failed`; now 404-only failures throw `all_not_found` with `JsonRpcErrorCode.NotFound` and recovery hint to verify IDs with `met_search`.
- **`kind` discriminator** on internal `FailItem` type — `'not_found' | 'error'` enables the 404-vs-network classification above.
- **Tests** — `all_not_found` path, `all_failed` network-error path, and mixed-failure path (some 404, some network) each covered independently; `hasCC0Image` verified in `format()` output.
